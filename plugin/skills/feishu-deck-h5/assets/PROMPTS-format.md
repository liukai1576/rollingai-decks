# PROMPTS.md — canonical format spec

`PROMPTS.md` is the **single source of truth** for the prompts-log
feature (see SKILL.md "Run-prompts log"). Multiple writers feed into it
(realtime agent appenders + post-hoc transcript adapters), but every
writer MUST emit the format below. Downstream analysis tools depend on
this stability.

## Why a canonical format

We want to support multiple agents (Claude Code, Codex, Mira, Cursor,
Aider, …) without analysis code multiplying per agent. The format is
the contract:

```
agent A → realtime append  ─┐
agent B → realtime append  ─┼─→ PROMPTS.md ─→ analyze-prompts.py
adapter (Claude Code) ─────┤   (canonical)    (one analyzer fits all)
adapter (Codex) ───────────┤
adapter (Mira) ────────────┘
```

If you add a new writer (a new agent or a new adapter), conform to this
spec. Don't invent variations — they break aggregation.

## File location

```
runs/<timestamp>-<slug>/output/PROMPTS.md
```

One PROMPTS.md per deck-run. NOT per session, NOT per day. A deck that
spans 3 conversations gets ONE PROMPTS.md with all 3 conversations'
prompts merged in chronological order.

## File structure

```markdown
# <Optional title — usually the deck's display name>

> Extracted **<N>** user prompts from **<M>** transcript(s) at
> <YYYY-MM-DD HH:MM>.
> Decks referenced: <slug1>, <slug2>, …
>
> Each entry: `## <timestamp>  <type>  <slide-refs>`.
> Types: `init / add-slide / bug-report / edit-slide / delete /
> confirm / meta` (keyword heuristic; see `classify()` in the adapter).

**By type**: meta=<n> · edit-slide=<n> · bug-report=<n> · …

## <YYYY-MM-DD HH:MM:SS>  `<type>` [<#slide-refs>]  [(agent: <id>)]
> <user prompt text, line by line, prefixed with `> `>
> <line 2 if multi-line>
>
> <blank line between entries — the `>` is REQUIRED on blank lines
> inside a single entry to keep markdown's block-quote unbroken>

## <YYYY-MM-DD HH:MM:SS>  `<type>` [<#slide-refs>]
> <next prompt>
```

Notes:

- **Title** (`# ...`) is optional but recommended. One per file.
- **Header block** (`> Extracted ...`) is generated by the writing
  tool. Include counts + decks referenced + timestamp of extraction.
- **By type** roll-up is a one-line summary. Omit if the writer
  doesn't compute it.
- **Each entry** starts with `## <timestamp>`. Timestamps are
  ISO-derived `YYYY-MM-DD HH:MM:SS` in local time (NOT UTC).
- **Type tag** in backticks: one of the seven types.
- **Slide refs** are space-separated `#NN` after the type, optional.
- **Agent tag** is `(agent: <id>)` after the slide refs, optional but
  recommended for realtime appenders. Identifies WHO wrote this entry
  (`claude-code-opus47`, `codex-cli`, `mira-agent`, …). Adapters
  may omit if reconstructing post-hoc.
- **Prompt body** is a markdown blockquote. Every line of the user's
  original prompt is preserved verbatim (including blank lines inside
  it — represented as `>` alone). NO rewriting, NO summarization, NO
  translation.

## Type taxonomy

| Type | Trigger | What it means |
|---|---|---|
| `init` | First deck-start prompt, "帮我做一个 deck" / "create deck" | Deck birth |
| `add-slide` | Leading "下一页" / "第 N 页" / "再做一页" / "新加一页" | New slide brief; bug-words inside this are CONTENT not complaint |
| `bug-report` | Visual / position / size / overflow / color complaint | **Highest analysis value — skill-gap signal source** |
| `edit-slide` | "改" / "换" / "调整" + slide reference | Iterative slide refinement |
| `delete` | "删" / "去掉" / "不要了" | Slide / element removal |
| `confirm` | Short OK / 加 / push / 同意 (regex match) | Approval to continue |
| `meta` | Fallback — discussion, design questions, no slide intent | Conversation-level (least analysis value) |

Classification is **heuristic by keyword scan**, not LLM. This is
intentional:

- Reproducible: same input → same tag, always
- Auditable: keyword lists are short and reviewable
- Tunable per-deck or per-org without retraining

If a class is consistently wrong, edit the keyword list and re-extract.
Do NOT introduce an LLM classifier — it makes the log less trustworthy
than the raw user text.

## Realtime-append contract (mandatory for any agent participating)

If you're an agent driving the feishu-deck-h5 skill, follow this:

**When**: after each user message, BEFORE generating any artifact in
response.

**What to append** (in this order, no exceptions):

```
\n## <ISO-local-time>  `<best-guess type>`<space-prefixed-slide-refs>  (agent: <your-id>)\n> <user message line 1>\n> <user message line 2>\n>\n
```

**Rules**:

1. **Verbatim**. Do NOT rewrite, summarize, translate, or "improve"
   the user's wording. If the user wrote "字小了，看不见", that's
   exactly what goes in. LLM remix = pollution.
2. **Best-effort type tag**. Use the same heuristic as the adapter
   (see `classify()` in `extract-from-claude-code.py`). It's fine if
   you guess wrong; downstream re-tagging is cheap.
3. **Idempotent**. If PROMPTS.md doesn't exist, create it with a `# `
   title header and the standard sub-headers. If it exists, append.
4. **No truncation**. Long prompts stay long. Don't `[...]`.
5. **Best-effort timestamp**. Use ISO-local-time `YYYY-MM-DD HH:MM:SS`.
6. **Agent ID**. Always include `(agent: <id>)` in realtime mode so
   later analysis can compare cross-agent prompt patterns.
7. **No fallback to a side channel**. If you can't file-write (e.g.
   sandboxed agent), at minimum print
   `PROMPT-LOG: <ts> | <type> | <verbatim>` to stdout so the user can
   paste it into PROMPTS.md by hand.

## Post-hoc adapter contract

Adapters that backfill PROMPTS.md from agent-specific transcripts
follow the same format. Conventions:

- Output filename: `extract-from-<agent>.py` (e.g.
  `extract-from-claude-code.py`, `extract-from-codex.py`)
- CLI signature:
  `python3 extract-from-<agent>.py <transcript-path>... [--filter-deck SLUG] [--filter-cwd PATH] [--out PATH] [--title STR]`
- Default to stdout if no `--out`
- Session-level deck filter: if ANY prompt in a transcript mentions the
  slug, include ALL prompts from that transcript (a conversation usually
  focuses on one deck; confirm-style follow-ups deserve to be kept)
- Don't include `(agent: <id>)` in post-hoc mode (we don't know which
  Claude Code version produced the entries unless the transcript
  records it; adapters that DO have this info SHOULD include it)
- Idempotent re-runs OK; either overwrite or merge — your choice, but
  document it in the adapter's header

## Anti-patterns (do NOT do these)

1. **LLM remix the user's prompt** — destroys the log's truthiness
2. **Filter out "trivial" prompts** like `OK` / `加` / `push` — they
   ARE signal (workflow cadence, iteration count)
3. **Include assistant responses** — out of scope; this log is the
   USER's voice, not the conversation
4. **Include tool outputs** — also out of scope
5. **Translate or normalize** Chinese → English (or vice versa) —
   keep verbatim; analysis tools handle multilingual matching
6. **Auto-collapse repeated entries** — repetition itself is signal
   ("user said 字小 5 times in a row" → red flag)
7. **Aggregate across multiple users automatically** — privacy
   boundary; aggregation is opt-in, manual collect to a separate repo

## Versioning

This is `PROMPTS.md format v1`. If the spec changes:

- Backward-compatible additions (new optional fields): keep version
- Breaking changes: bump to v2 + provide a migration tool

Writers SHOULD NOT emit a version line — readers assume v1 and tolerate
unknown optional fields gracefully.

## Open questions (future work)

- **Privacy / scrubbing**: how to handle PII in user prompts? For now,
  user responsibility. Future: optional `--scrub` flag to redact
  email / phone / customer-name patterns before writing.
- **Cross-deck analysis tooling**: `analyze-prompts.py` (Phase 2) reads
  multiple PROMPTS.md files and rolls up by keyword frequency. Spec
  TBD until Phase 1 generates enough data to design against.
- **Agent ID standardization**: needs a registry once we have ≥ 2
  agents emitting realtime entries.
